# Codex 工作台审计契约

本文件是三个体检引擎、证据等级、项目状态和最终输出的唯一产品契约。新增检查项前，先确定它属于哪个引擎、使用什么证据、能支持什么结论，以及写入最终报告的哪个位置。

## 产品边界

把本 Skill 定义为个人 Codex 工作台审计器：在用户批准的范围内，检查协作方式、Codex 配置与指令体系，以及可识别项目的执行状态，再恢复一份可继续执行的行动顺序。

- 默认只读，不自动修改配置、规则、Skills、聊天或项目。
- 只审计指定时间、目录和可读取来源，不声称覆盖电脑上的所有项目或所有行为。
- 不给用户、模型或项目价值打分。只报告可观察行为、影响和改进方法。
- 不把“没有观察到”写成“不存在”，不把助手自称完成写成项目已完成。
- 行动顺序是带理由的建议；截止时间、当前目标和是否继续项目由用户决定。

## 共同处理流程

三个引擎共用同一条处理链：

```text
范围与授权
  -> 确定性采集
  -> 脱敏与证据索引
  -> 跨来源关联
  -> 语义诊断
  -> 状态与优先级判断
  -> 统一报告
```

确定性脚本负责配置解析、文件存在性、Git、会话统计和清单。Codex 负责阅读少量必要上下文、比较来源和解释根因。语义判断不得补造脚本没有采集到的事实。

## 三个体检引擎

### 1. 交互协作引擎

回答“我和 Codex 的协作方式，哪里正在浪费时间，哪些做法值得保留”。

输入：

- 指定时间范围内的会话消息、工具调用和工具结果
- 顺利完成样本与摩擦样本
- 安装的 Skill 清单、触发描述和可识别调用证据
- 与任务相关的 AGENTS.md、测试和交付证据

输出：

- 可复用的成功协作模式
- 返工、范围扩大、过早停止、缺少验证、降级交付和任务切换等弯路
- 用户输入、Codex 执行、共同流程或环境工具的主要归因
- Skill 的 `confirmed`、`probable`、`missed_opportunity`、`not_observed` 或 `unknown` 使用状态
- 应放入本次 prompt、用户 AGENTS.md、项目 AGENTS.md、Skill、配置或自动化检查的改进动作

限制：

- `not_observed` 必须同时显示时间范围、任务数量和可解析会话数量。
- 没有适用任务时，不能把 Skill 标为漏用或建议删除。
- Skill 工作流不完整必须有调用证据和缺失步骤证据，不能只根据最终结果推测。
- 不复制长聊天原文，不从单次纠正建立全局规则。

### 2. Codex 工作台引擎

回答“配置、规则和扩展之间，有没有冲突、失效、重复或作用域错误”。

输入：

- 用户级、项目级和可识别 profile 配置
- 用户级、项目级及嵌套 AGENTS.md
- 用户级和项目级 Skills、MCP 与可确认启用状态的插件
- 会话中可观察到的规则遵守或违反证据

输出：

- 配置和指令继承关系图
- 权限、沙箱、网络、MCP 和敏感信息风险
- AGENTS.md 的作用范围、可执行命令、完成标准、验证方式、时效性和重复内容
- Skill 完整性、触发清晰度、语义重叠和冲突流程
- `明确冲突`、`作用域错位`、`重复`、`潜在张力` 或 `疑似过期` 五类关系

限制：

- 只有一条规则要求某行为、另一条明确禁止该行为时，才标记明确冲突。
- 相似表述不等于冲突；无法确定优先级时标记潜在张力。
- 插件目录存在不等于插件已启用，Skill 安装不等于实际触发。
- “Codex 未遵守规则”必须同时引用适用规则和对应行为证据。

### 3. 项目恢复引擎

回答“审计范围内有哪些项目、分别做到哪一步、哪些事项没有闭环、下一步如何衔接”。

项目只从以下来源发现：

- 会话中的工作目录
- Codex 可读取的最近项目记录
- 用户明确指定的目录
- 已确认属于这些项目的配置或工作区记录

不得未经授权扫描整块磁盘。报告必须列出发现来源、检查成功数量和无法访问数量。

每个项目检查：

- 目标、最近活动时间、Git 分支和工作区状态
- 已交付且经过验证的内容
- 已实现但待验证、待提交、待发布或待验收的内容
- 计划未实现、聊天承诺未落地、TODO 与实际代码不一致的事项
- 当前错误、外部依赖、用户决策和其他阻塞
- 下一步任务、依赖关系和关闭条件

限制：

- 项目状态至少由聊天、文件/Git、测试/构建/验收三类来源中的一类直接证据支持；“已完成”必须满足更严格条件。
- 不用投入时间推断商业价值，不替用户决定项目是否继续。
- 没有足够目标或现场证据时标记状态不明，不猜测完成百分比。

## 证据等级

证据等级描述“依据是什么”，置信度描述“结论有多稳”，两者必须分开记录。

| 等级 | 名称 | 定义 | 允许的表述 |
| --- | --- | --- | --- |
| `A` | 已验证 | 当前审计直接读取的结构化配置、文件、Git、测试、构建或交付结果 | “已确认”“当前存在” |
| `B` | 已佐证 | 至少两个独立来源指向同一结论，例如聊天承诺与仓库产物相符 | “证据共同表明” |
| `C` | 单源观察 | 单条聊天、单个间接信号或尚未交叉验证的语义判断 | “观察到”“可能” |
| `D` | 范围内未观察 | 在明确时间、任务数和覆盖率内未发现某行为 | “在本次范围内未发现” |
| `U` | 无法判断 | 来源缺失、格式无法解析或证据互相矛盾 | “暂无法判断” |

约束：

- `A` 通常为高置信度，`B` 为高或中，`C` 为中或低，`D` 最高只能为中，`U` 不生成确定结论。
- 所有 `D` 级结论必须显示时间窗口、分母和覆盖缺口。
- 私有聊天片段只保存在私有证据包；公开报告使用证据编号和概括。
- 结论升级时保留原证据链，例如从 `C` 升为 `B` 时记录新增来源。

## 项目状态机

项目只允许使用以下用户可见状态：

| 状态 | 进入条件 |
| --- | --- |
| `unknown` 状态不明 | 缺少目标、现场或时间证据，无法可靠分类 |
| `planned` 尚未开始 | 有明确目标或计划，但没有实现、产物或执行记录 |
| `in_progress` 进行中 | 已有实际修改或产物，且仍有明确未完成工作 |
| `verification_pending` 部分完成/待验证 | 核心实现存在，但测试、构建、提交、发布、文档或验收中的关键步骤缺失 |
| `blocked` 被阻塞 | 有明确未解决依赖、错误或用户决策，并且它阻止下一关键步骤 |
| `stale` 疑似遗忘 | 存在未完成事项，超过报告声明的闲置阈值没有继续，也没有明确暂停或关闭决定 |
| `completed` 已完成 | 有目标对应的交付物，并有测试、构建、验收或用户确认，且没有已知关键未完成项 |
| `archived` 已归档 | 用户明确停止、替代或归档，后续不再进入默认行动队列 |

允许的主要转换：

```text
unknown -> planned -> in_progress -> verification_pending -> completed -> archived
                         |                   |
                         +----> blocked <----+
                         |                   |
                         +----> stale -------+

blocked/stale -> in_progress
completed -> in_progress    （出现新的明确范围）
任意状态 -> archived        （用户明确决定）
```

状态判断规则：

- 助手自称“完成”只能作为 `C` 级观察，不能单独进入 `completed`。
- 用户最后追加任务而没有后续交付时，保留为未完成事项。
- `stale` 必须显示闲置阈值和最近活动时间，只能表述为“疑似遗忘”。
- 状态冲突时选择更保守状态，并在报告中列出冲突证据。

## 行动排序

不使用不可解释的总分。先按以下层级排序，再在同层内比较截止时间、下游影响、证据强度和预计收尾成本：

1. 用户当前明确目标和截止时间
2. 正在阻塞其他工作的任务
3. 补少量工作即可形成可验证交付的任务
4. 已有投入且疑似遗忘、需要用户决定继续或关闭的项目
5. 会持续制造返工的配置、规则和协作流程
6. 尚未开始的全新项目和功能

每个行动必须包含排序理由、前置依赖、完成条件、来源发现和是否需要用户批准。缺少用户目标或截止时间时，先展示建议顺序并明确请求确认，不把建议写成命令。

## 统一输出结构

最终生成一个 `health-check.md`，包含四份用户结果；基础脚本的 `report.md` 和 `report.json` 作为确定性输入保留。

```markdown
# Codex 工作台体检

## 审计范围与覆盖
[时间、项目来源、读取成功/失败、隐私边界]

## 一句话结论
[最值得保留的模式、最需要处理的问题、最优先行动]

## 1. 协作诊断
[成功模式、协作弯路、Skill 使用观察、优化动作]

## 2. 配置诊断
[配置关系图、AGENTS.md 关系、Skill/MCP 问题和冲突]

## 3. 项目恢复地图
[每个项目的状态、已完成、未完成、阻塞和下一步]

## 4. 建议行动顺序
[排序后的任务、理由、依赖、完成条件和审批边界]

## 证据边界
[A/B/C/D/U 分布、无法读取来源、不能判断事项]
```

所有发现使用同一字段：

| 字段 | 要求 |
| --- | --- |
| `finding_id` | 稳定编号 |
| `engine` | `collaboration`、`workbench` 或 `projects` |
| `observation` | 观察到的事实或模式 |
| `impact` | 对时间、质量、安全或注意力的影响 |
| `priority` | `P0` 至 `P3` |
| `evidence_grade` | `A/B/C/D/U` |
| `confidence` | 高、中或低 |
| `evidence_refs` | 脱敏证据编号，不放长原文 |
| `coverage` | 时间、样本、目录和解析覆盖 |
| `recommendation` | 最小可执行改进 |
| `placement` | prompt、用户/项目 AGENTS.md、Skill、config、测试或项目任务 |
| `approval_required` | 是否需要用户批准 |
| `verification` | 改进后怎样复测 |

## 验收条件

一次完整体检只有同时满足以下条件才算完成：

- 三个引擎分别报告 `complete`、`partial`、`unavailable` 或 `failed`，不隐藏覆盖缺口。
- 每个结论都有证据等级、置信度、证据引用和覆盖范围。
- 每个项目都有允许状态之一；状态不明时明确缺什么证据。
- 四个最终结果齐全，行动项可以追溯到发现和用户目标。
- 公开报告不包含密钥、完整私人路径或长聊天原文。
- 未经用户批准，没有修改配置、规则、Skills、聊天或项目。
