# 检查规则与判断口径

## 基本原则

每个发现都包含：检查域、稳定规则编号、P0-P3 优先级、置信度、证据、建议和是否需要批准。规则只在证据达到最低样本量后触发。

## 配置

- `CFG001`：`danger-full-access` 与 `approval_policy = never` 同时出现。权限与审批同时放宽，误操作缺少第二道边界。
- `CFG002`：单独使用 `approval_policy = never`。这可能适合自动化，但交互开发需要确认是否符合用户意图。
- `CFG003`：配置键名疑似密钥且值直接写入文件。只报告键路径，不报告值。
- `CFG004`：使用官方已标记过时或保留的字段。
- `CFG005`：配置中的项目目录不存在。只报告数量，不泄露完整路径。
- `CFG006`：MCP 通过通用 shell 启动。该方式会扩大命令拼接和环境展开范围，需要人工确认必要性。

## Skills

- `SKL001`：缺少有效 frontmatter、`name` 或 `description`。
- `SKL002`：目录名与 Skill 名明显不一致，可能造成维护和分发混乱。
- `SKL003`：描述没有说明触发场景，隐式调用可靠性不足。
- `SKL004`：多个 Skill 使用相同名称或相同描述。先确认作用域与真实路径，不能直接删除。
- `SKL005`：`SKILL.md` 超过 500 行。建议把领域资料下沉到 references，而不是机械缩短必要规则。
- `SKL006`：相对 Markdown 资源链接不存在。
- `SKL007`：Skill 或脚本出现远程脚本直管道执行、宽泛破坏性删除或读取常见密钥文件的指令。

## 聊天协作

- `SES001`：至少 10 条用户消息且高置信返工信号比例达到 15%。这只能证明沟通发生返工，不能单独归因于用户或模型。
- `SES002`：至少 12 条用户消息且“继续/确认”类消息达到 25%。建议在任务开头授权连续执行并定义完成条件。
- `SES003`：至少 5 个会话，40 条以上用户消息的长会话占比超过 20%。建议把新目标拆到新会话，同时保留同一目标的连续上下文。
- `SES004`：可识别工具结果至少 10 个且失败信号达到 20%。需要结合具体项目日志复核，不能只看关键词。

## 项目执行

- `PRJ001`：当前目录具有代码项目特征但没有 Git 仓库。
- `PRJ002`：仓库尚无提交，或未跟踪/已修改文件数量过大，当前成果缺少清晰恢复点。
- `PRJ003`：项目有持续 Codex 会话但没有项目级 AGENTS.md。只有出现重复工作时才建议沉淀规则。
- `PRJ004`：AGENTS.md 过长。优先删除重复与无效规则，或将局部规则移到更近的目录。
- `PRJ005`：当前项目的聊天返工信号显著高于总体样本。该项只提示复盘，不能直接判定技术路线错误。

## 不做的推断

- 不根据 Skill 数量直接认定上下文浪费。
- 不根据会话长度直接认定项目失败。
- 不根据一次否定直接认定提示词差。
- 不根据插件目录存在就认定插件启用。
- 不根据没有 AGENTS.md 就认定项目管理不规范。
