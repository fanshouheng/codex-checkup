# Codex 协作诊断框架

## 目标

把少量顺利完成样本和摩擦样本转化为可执行的协作改进。诊断协作系统，不评价用户或模型的能力、态度和人格。

统计报告只能定位候选，不能解释根因。先分别阅读顺利样本和摩擦样本，再比较两类对话在任务定义、执行连续性、证据、验证和交付上的差异。至少阅读同类模式的两个样本再下稳定结论；只有一个样本时标记为“单次事件”。

## 顺利完成样本

`successful_completion` 表示会话中没有明显纠正或反复推动，最终回复明确完成目标，并存在可关联的成功测试、构建或其他工具结果；其依据标记为 `completion_and_verified_tool`。用户明确确认结果正确时也可作为成功候选。它们仍不是自动判定：阅读上下文后确认目标确实闭环，而不是只因为助手说“完成”就采信。

从顺利样本提取：

- 用户输入中哪些目标、边界、素材或验收信息帮助了执行
- Codex 是否正确保持范围并连续推进
- 哪些检查、工具和交付格式构成了可信闭环
- 哪些做法只适用于单个项目，哪些可以跨项目复用

## 六个诊断维度

| 维度 | 典型信号 | 需要区分的根因 | 理想状态 |
| --- | --- | --- | --- |
| 目标与完成条件 | 多次说“不对”“不是这个” | 初始目标缺失、Codex 误读、完成标准未定义 | 开始前能复述目标、边界和可验证完成条件 |
| 范围控制 | “只改这里”“其他别动” | 用户未说明边界、Codex 擅自扩大改动、反馈后全量重写 | 每个变更都能追溯到请求，反馈采用增量修改 |
| 自治程度 | 用户反复说“继续”、Codex 频繁停下 | 授权范围不清、Codex 过度确认、存在真正的产品分歧 | 可逆且范围内的步骤连续完成，只在实质分歧或高影响动作前提问 |
| 证据纪律 | “你确定吗”“不要瞎编”“先看源码” | Codex 未检查实际文件、信息不足却给确定结论、用户缺少必要材料 | 先列已确认事实和缺失信息，结论能引用本地证据 |
| 验证与交付 | “你测试了吗”“还没完成” | Codex 把写代码当完成、没有运行验证、只给建议未交付文件 | 交付前运行与风险相称的验证，明确已完成与未验证部分 |
| 上下文与持久规则 | 同一纠正跨会话重复、目标中途切换 | 新目标留在旧会话、项目知识未进入 AGENTS.md、流程知识未进入 Skill | 一个会话围绕一个目标，重复纠正进入最小持久规则 |

## 责任归类

每个模式选择一个主要归类，不要把所有问题都归因于提示词：

- **用户输入可优化**：关键目标、对象、边界或完成条件在开始时确实缺失。
- **Codex 执行需优化**：输入已经足够，但 Codex 扩大范围、未读真实文件、停止过早或未验证。
- **共同流程需优化**：单轮输入无法解决，需要明确阶段、决策点、验证门槛或交接方式。
- **环境/工具需优化**：主要问题来自权限、依赖、路径、MCP、Git 或运行环境，而非对话方式。

证据不能区分时写“暂不能归因”，并说明还需要什么。

## 建议放置位置

不要把所有建议都塞进全局提示词：

- 当前一次任务特有：放在本次 prompt。
- 同一项目反复出现：放在项目 `AGENTS.md`，保持短小。
- 跨项目反复出现的个人偏好：放在全局 `AGENTS.md`。
- 稳定的多步骤专业流程：放在 Skill。
- 权限、模型、MCP 或 sandbox：放在 `config.toml`，修改前需要用户批准。
- 可以机械验证的质量门槛：优先交给测试、lint、hook 或 CI，而不是只写自然语言规则。

形成建议前读取 [codex-practice-network.md](codex-practice-network.md)。每项建议绑定至少一个 `PRAxxx` 节点和建议依据；没有匹配节点时标记为本地推导。X 上的传播量、认证标记或单个实践者经验不能单独升级成全局规范。

## 质量要求

- 不输出“提示词写清楚一点”“多沟通”“加强验证”这类空泛建议。
- 每个建议都给一个可直接采用的文本、动作或门槛。
- 不复制大段聊天原文。优先概括，只在区分根因确有必要时引用一句短语。
- 不因一次纠正建立全局规则；跨项目重复才考虑全局规则。
- 不把 Codex 本应完成的检查转嫁给用户补充提示。
- 不把“助手自称完成”直接当作成功；确认请求目标、产物和验证是否对应。
- 不只写问题。至少提炼一个顺利样本中的有效模式；没有可靠顺利样本时明确说明。
- 不输出没有 `PRAxxx` 或本地推导链的通用建议；区分官方确认、实践支持和实验性能力。
