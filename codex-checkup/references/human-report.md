# 面向人的报告协议

主报告的读者是想改善 Codex 使用方式的人，不是审计器开发者。内部字段可以保证可信度，但不能占据用户首先看到的内容。

## 两层输出

一次体检生成两份语义报告：

- `health-check.md`：面向用户的主报告。先回答哪里反复绕路、为什么、怎样改、哪些值得做成 Skill。
- `health-check-evidence.md`：技术证据附录。保存覆盖范围、模块状态、证据等级、规则编号、实践节点、完整发现台账和无法判断项。

基础脚本生成的 `report.md`、`report.json` 和私有证据包继续保留。主报告不复制这些文件的内部结构。

## 主报告写作规则

1. 第一屏先给最重要的三个用户结论，不先解释扫描了多少文件。
2. 用“你现在的流程 / 为什么会返工 / 改成什么流程”表达，不用“引擎、字段、命中规则”表达。
3. 每个稳定问题都说明出现次数或涉及项目数；无法量化时间时写返工类型，不猜小时数。
4. 重复次数必须注明任务清单覆盖范围；清单被截断时不能声称没有其他重复流程或 Skill 候选。
5. 用一两个脱敏项目例子解释模式，但不复制聊天原文。
6. 每个问题给出下一次可直接采用的开场文本或 Codex 行为规则。
7. 明确建议应放在本次任务、`AGENTS.md`、现有 Skill、新 Skill、自动检查或配置中的哪一层。
8. 主报告只列最值得关注的进行中、待验证、阻塞或疑似遗忘项目，默认不展开大量状态不明项目。
9. 主报告控制在大约 1500-4000 个中文字；稳定问题最多 5 个，重点项目最多 8 个。

主报告正文不要出现以下内部表达：

- `complete`、`partial`、`A/B/C/D/U`、`P0-P3`
- `PRAxxx`、`SKLxxx`、`PRJxxx`、`finding_id`
- `engine`、`recommendation_basis`、`stanza`、`fixture`
- `$PROJECT`、完整报告路径、哈希标签和大段工具统计

`AGENTS.md`、Skill、MCP、Git、测试等用户需要采取行动的真实对象可以保留。必要的可信度使用“已确认 / 多个样本共同出现 / 单次观察 / 暂时无法判断”这类自然语言。

## Skill 候选判断

不要因为任务重复两次就立刻新建 Skill。依次判断：

1. **重复性**：至少两个独立任务或项目出现，或者用户明确说明这是长期高频工作。
2. **稳定性**：触发条件、输入、核心步骤和交付物能够说清楚。
3. **流程长度**：通常包含三个以上步骤、多个工具或需要固定判断顺序。
4. **可验证性**：可以定义成功结果，不只是“感觉更好”。
5. **已有能力**：先检查已安装 Skill；已有能力覆盖时优先优化或组合，不创建重复 Skill。

分流规则：

- 一句长期偏好或行为边界：写入合适作用域的 `AGENTS.md`。
- 单个项目的一次决策：留在当前任务或项目计划。
- 可以机械判断的质量门槛：做成测试、lint、hook 或脚本。
- 模型、权限、MCP 和网络开关：放入配置。
- 稳定、多步骤、会被自然语言反复调用的工作流：适合做成 Skill。
- 证据只来自一次事件：标记“先观察”，不创建新 Skill。

“没有可靠的新 Skill 候选”是有效结论。不要为了让报告显得有价值而硬造 Skill。

## 复测报告

复测主报告仍然先面向用户：

1. 哪个具体问题真的解决了。
2. 哪些协作流程仍然反复出现。
3. 哪些新问题值得关注。
4. Skill 候选判断是否因新任务样本发生变化。

基线版本、模块口径、规则变化、回归用例和完整量化对照全部放进 `health-check-evidence.md`。主报告不要以“审计范围与覆盖”开头。

## 技术证据附录

`health-check-evidence.md` 保留：

- 时间窗口、会话/项目/文件覆盖与缺口
- 各模块状态和基础报告链接
- 证据等级、置信度、发现编号和实践节点
- 完整发现台账和完整项目恢复表
- Skill 候选所用任务清单范围、未覆盖数量和反例
- 复测时的基线版本、规则变化和逐项证据对照

附录仍需脱敏。主报告中的每个稳定结论都应能在附录中找到对应证据，但用户不需要先读附录才能理解建议。
