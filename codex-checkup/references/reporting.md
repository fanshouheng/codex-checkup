# 报告解释指南

## 优先级

- `P0`：敏感信息暴露、明显破坏性行为等应立即处理的风险
- `P1`：高概率影响安全、返工或交付的风险
- `P2`：应安排处理的流程、维护或效率问题
- `P3`：低风险观察项，先验证再行动

## 置信度

- `高`：来自结构化配置、文件存在性或 Git 等可直接验证事实
- `中`：来自稳定但不完备的模式和足够样本
- `低`：本地格式不稳定、样本少或只能进行间接推断

证据等级与置信度必须同时给出。`A/B/C/D/U` 的定义、项目状态和统一字段以 [audit-contract.md](audit-contract.md) 为准。

## 怎样给建议

先处理少量高影响问题。每项建议应说明：先做什么、保留什么能力、如何验证、是否需要批准，并引用 [codex-practice-network.md](codex-practice-network.md) 中的 `PRAxxx` 和建议依据。没有项目级证据时，不要说项目“路子错了”；应说当前证据只能支持哪一种流程风险。

基础 `report.md` 和 `report.json` 负责结构化配置、Skills、统计和项目事实；最终 `health-check.md` 负责合并协作诊断、配置诊断、项目恢复地图和建议行动顺序。不要用基础统计报告替代语义诊断，也不要把私有片段复制进公开报告。

报告交付不是流程终点。最终回复先给报告链接和核心结论，再推荐一个最值得先处理的真实行动，并给出五个入口：开始第 1 项、只优化协作、只优化配置、只恢复项目、暂不修改。用户选择后按 [remediation.md](remediation.md) 继续，不重新讲一遍体检结果。

## 覆盖状态

- `complete`：该模块按预期读取并完成检查
- `partial`：只读取到部分来源或遇到未知格式
- `unavailable`：来源不存在、权限不足或运行环境不支持
- `failed`：模块执行错误，报告应保留错误类别但不泄露敏感数据
