# 整改与复测协议

本文件定义体检之后怎样继续执行。目标不是自动修改一切，而是让每项建议变成有边界、有授权、有验证、可跨任务恢复的行动。

## 找到基线

1. 优先使用用户明确指定的 `health-check.md`。
2. 未指定时，使用当前项目中最近修改且结构完整的 `health-check.md`。
3. 同时存在多个候选且范围不同，先展示日期和范围，请用户选择；不要静默猜测。
4. 读取配套的 `health-check-evidence.md` 和其中引用的 `report.json`，保留 `days`、`project`、模块、覆盖状态和发现编号，供复测使用。
5. 基线缺失、关键路径失效或范围无法恢复时，说明原因并重新体检，不伪造整改状态。

`health-check.md` 是审计时点的不可变基线。不要为了打勾而改写原报告。

## 整改状态

用户批准开始整改时，创建或恢复 `.codex-health-private/remediation-state.json`。状态文件不得保存聊天原文、密钥、完整私人路径或配置值。

```json
{
  "schema_version": 1,
  "source_report": "health-check.md",
  "baseline": {
    "days": 30,
    "project": "$PROJECT",
    "modules": ["config", "skills", "sessions", "portfolio", "project"]
  },
  "actions": [
    {
      "action_id": "ACT-001",
      "finding_ids": ["WB-001"],
      "title": "修复安全检测误报",
      "status": "pending",
      "approval_required": true,
      "target": "Skill scripts/tests",
      "completion_condition": "误报消失且真实危险样本仍命中",
      "verification": [],
      "retest_outcome": "not_run",
      "notes": []
    }
  ]
}
```

`status` 只使用：

- `pending`：尚未开始
- `awaiting_approval`：范围已说明，等待必要授权
- `in_progress`：已开始修改
- `verified`：目标验证通过，但尚未同口径复测
- `blocked`：存在明确阻塞
- `failed`：修改或验证失败，保留现场与原因
- `skipped`：用户决定暂不处理

`retest_outcome` 只使用 `not_run`、`resolved`、`persisting`、`regressed` 或 `not_comparable`。执行状态和复测结论分开记录，避免把“代码改完”误写成“问题已解决”。

## 单项执行循环

每次只推进一个边界清楚的行动：

1. **核验**：重新检查来源发现仍然成立；误报或已过期时更新状态，不继续修改。
2. **定界**：列出目标、预计修改位置、保留能力、风险和验证方法。
3. **判断授权**：用户点名行动即授权可逆的限定范围修改；高影响动作仍单独确认。
4. **建立恢复点**：Git 已跟踪文件依靠差异审阅；Git 外配置先创建时间戳备份。不要把密钥复制进报告或状态文件。
5. **执行**：做最小修改，不顺带清理其他问题。
6. **验证**：运行与完成条件直接对应的测试、解析、连通性或人工验收。失败时记录真实结果，不降级宣称完成。
7. **更新状态**：保存修改位置、验证结果和剩余风险；然后推荐下一项。

以下动作即使在“全部优化”请求中也需要单独确认：

- 删除 Skill、聊天、配置项、项目或用户数据
- 修改凭据、权限、沙箱、网络或全局 MCP
- 初始化 Git、提交、推送、合并或重写历史
- 归档、关闭或放弃项目
- 跨出用户指定项目和目录的修改

## 报告后的交互入口

体检完成后的回复不能只给报告链接。使用当前报告中的真实行动名称，给出：

```text
建议先处理：第 1 项「[行动名称]」，因为 [一句排序理由]。

下一步：
1. 开始第 1 项
2. 只优化协作方式
3. 只优化配置、AGENTS.md 和 Skills
4. 只恢复未完成项目
5. 暂不修改，保留报告
```

用户回复数字、“继续优化”“按顺序处理”或“修复第 N 项”时，直接恢复整改模式，不重新解释整个产品。

## 同口径复测

整改批次完成或用户要求复测时：

1. 从状态文件恢复原 `days`、`project`、模块和覆盖条件。
2. 运行相同的确定性扫描；需要语义判断时使用相同抽样规则。
3. 对每个来源发现比较证据，而不是只比较问题总数。
4. 输出四类结果：`resolved`、`persisting`、`regressed/new`、`not_comparable`。
5. 保留成功模式，说明哪些指标不能直接比较以及原因。
6. 更新状态文件，但不覆盖原始 `health-check.md`。

复测完成后的摘要优先回答：改了什么、验证了什么、问题是否真的消失、下一步是否仍值得继续。
