# Codex 实践知识网络

核验日期：2026-07-14。

本文件把官方 Codex 规范、OpenAI 官方 X 动态和具名开发者的实战经验，转成体检时可调用的“症状 -> 实践 -> 放置位置 -> 验证”知识节点。它用于解释用户应该怎样改进，不用于替代对用户本地证据的检查。

## 使用规则

1. 先用本地会话、配置、Git、文件和测试确定观察事实，再选择实践节点。
2. 本地事实使用审计契约的 `A/B/C/D/U` 证据等级；知识来源使用本文件的权威等级，两者不能混用。
3. 每项建议至少记录一个 `PRAxxx` 节点；没有匹配节点时写明“本地推导”，不要伪造官方规范。
4. 同一症状可能有不同根因。先区分用户输入、Codex 执行、共同流程和环境工具，再选建议。
5. 社区技巧只有在适用条件满足时才能建议。官方文档与社区说法冲突时，以当前官方文档和当前环境可验证行为为准。
6. 不复制长帖子或整套他人配置，只保存可验证的原则、来源链接和适用边界。

## 来源权威等级

| 代码 | 来源 | 能支持什么 | 不能支持什么 |
| --- | --- | --- | --- |
| `O` | 当前 OpenAI/Codex 官方文档 | 产品行为、配置规则和正式最佳实践 | 用户本地是否真的遵守 |
| `OX` | OpenAI 官方 X 账号 | 新能力发布、官方示例和产品方向 | 尚未进入文档的稳定接口承诺 |
| `P` | 具名实践者的具体工作流、代码或复盘 | 有条件的实战建议和失败模式 | 对所有用户普遍成立的规范 |
| `C` | 普通社区帖子或二次总结 | 待验证线索、用户语言和常见误区 | 直接写入全局规则的依据 |
| `R` | 被官方边界或反证否定的流行说法 | 反例和风险提醒 | 正向建议 |

建议依据使用四种状态：

- `official-confirmed`：有 `O` 来源直接支持。
- `official-plus-practice`：官方原则与具名实践者经验互相佐证。
- `practice-supported`：只有具名实践者证据，必须写适用条件。
- `experimental`：只有官方 X 新能力或社区线索，使用前重新确认当前环境。

## 诊断路由

| 本地症状 | 先检查 | 首选节点 |
| --- | --- | --- |
| 多次说“不对”“不是这个” | 初始目标、输入材料、边界和验收是否明确 | `PRA001`、`PRA002`、`PRA015` |
| 反复说“继续”或 Codex 过早停下 | 是否定义成功证据、迭代方式和阻塞条件 | `PRA001`、`PRA003` |
| 写了代码但没有测试或交付 | 是否存在可运行验证、Git 差异和交付物 | `PRA003`、`PRA007`、`PRA020` |
| 会话越来越长、混入多个目标 | 目标是否真的分支，主线程是否被日志污染 | `PRA004`、`PRA005` |
| 多个代理相互冲突 | 子任务是否独立、是否同时写相同目录 | `PRA005`、`PRA006`、`PRA017` |
| Codex 修改了无关代码或过度设计 | 请求边界、最小实现和差异审查是否存在 | `PRA015`、`PRA020` |
| 同类要求跨会话重复 | 规则是个人偏好、项目规则还是目录规则 | `PRA008`、`PRA009` |
| Skill 没触发或流程只执行一半 | 描述、正反例、输入输出和脚本契约 | `PRA010`、`PRA011`、`PRA012` |
| 重复手工做稳定流程 | 流程是否可检查、可编辑、输入输出稳定 | `PRA010`、`PRA012`、`PRA013` |
| 项目缺少恢复点或并行任务互相覆盖 | Git 状态、分支、提交和 worktree 条件 | `PRA006`、`PRA007` |
| 自动维护任务经常跑偏 | 项目愿景、自治门槛和验证路径是否明确 | `PRA014`、`PRA019` |
| 工具失败、输出难以判断 | 工具是否可组合、输出是否确定、权限是否过宽 | `PRA016`、`PRA018` |

## 实践节点

### PRA001 强目标契约

- **适用**：长任务、持续迭代、用户反复推动、完成定义不一致。
- **做法**：同时定义结果、验证面、约束、允许边界、迭代策略和阻塞停止条件。目标模糊时，先让 Codex 起草目标，再由用户收紧关键边界。
- **放置**：当前 prompt 或 `/goal`；不要默认写进全局 `AGENTS.md`。
- **验证**：最终回复能逐项对应完成证据、未完成项和阻塞条件。
- **依据**：`O01`、`X09`，`official-plus-practice`。

### PRA002 提供必要上下文与示例

- **适用**：Codex 根据猜测补齐业务背景、输出格式多次不符合预期。
- **做法**：指出真实文件、当前数据、约束来源和一个正确输出示例；不要把整个项目无差别塞进 prompt。
- **放置**：当前 prompt；稳定样例可放 Skill references 或项目文档。
- **验证**：首次输出使用了指定材料，且格式返工下降。
- **依据**：`O02`，`official-confirmed`。

### PRA003 用结果反馈环驱动执行

- **适用**：用户逐步遥控、Codex 写完即停、优化过程没有客观反馈。
- **做法**：给成功标准，让 Codex先建立测试或基线，再在测试、基准、浏览器或真实产物反馈中循环。
- **放置**：任务 prompt、项目测试或 Skill 工作流。
- **验证**：每轮都有可比较结果，最终结论来自测试而不是助手自述。
- **依据**：`O01`、`X03`，`official-plus-practice`。

### PRA004 一个任务保持一个连贯目标

- **适用**：会话混入多个独立需求，旧决策与新目标互相干扰。
- **做法**：同一问题留在同一任务中保留推理链；只有目标真正分支时才 fork 或新建任务，并写交接摘要。
- **放置**：个人协作协议。
- **验证**：新任务能说明继承的结论与不继承的范围，原任务仍可恢复。
- **依据**：`O09`，`official-confirmed`。

### PRA005 子代理承担有边界的读密集工作

- **适用**：主线程被探索记录、日志和测试输出淹没，或存在可独立并行的分析。
- **做法**：把探索、测试、分类、日志分析和摘要交给子代理；主线程保留需求、决策和最终输出，只接收摘要。
- **放置**：当前任务编排或稳定 Skill 工作流。
- **验证**：子任务输入、输出和结束条件明确，主线程没有原始日志堆积。
- **依据**：`O03`，`official-confirmed`。

### PRA006 独立写任务使用 worktree 隔离

- **适用**：同一 Git 项目存在多个真正独立的写任务，需要并行而不污染本地工作区。
- **做法**：每个独立任务使用单独 worktree/分支；需要本地 IDE 或唯一运行环境时使用 Handoff 回到 Local。
- **放置**：任务启动方式和项目 Git 流程。
- **验证**：任务修改位于独立 checkout，分支不被两个 worktree 同时占用，合并前分别验证。
- **依据**：`O04`、`X10`，`official-plus-practice`。

### PRA007 把 Codex 输出当作 PR 审查

- **适用**：工作区长期脏乱、修改难以回滚、助手声称完成但差异未检查。
- **做法**：委派前保持 Git 状态清楚；使用功能分支或隔离工作区；检查完整 diff、运行针对性验证并做小步提交。
- **放置**：项目 `AGENTS.md`、测试流程或 Git hook，不要只靠口头提醒。
- **验证**：每个提交范围清晰、可回滚，交付包含测试和 diff 复核。
- **依据**：`O05`、`X03`，`official-plus-practice`。

### PRA008 按作用域放置 AGENTS.md

- **适用**：全局与项目规则冲突、局部要求污染其他仓库、同一规则重复出现。
- **做法**：跨仓库个人原则放用户级；项目命令与约束放根目录；子目录专属规则放更近的嵌套文件。理解 override、合并顺序和大小上限。
- **放置**：用户、项目或嵌套 `AGENTS.md`。
- **验证**：从当前目录重建指令链后，每条规则只有一个合理归属，近端规则没有意外覆盖上层安全边界。
- **依据**：`O06`，`official-confirmed`。

### PRA009 从重复证据提炼规则，不照抄万能模板

- **适用**：全局指令很长、包含陌生工作流，或者热门模板与项目实际矛盾。
- **做法**：只沉淀跨任务重复出现且仍有效的偏好、命令和验收方式。具体规则应解决观察到的问题，例如注释面向什么读者，而不是复制整套他人配置。
- **放置**：由作用域决定的 `AGENTS.md`；临时要求仍留在 prompt。
- **验证**：能为每条新增规则指出至少两个相关事件或一个明确项目事实；删除后无行为影响的规则进入清理候选。
- **依据**：`O06`、`X08`、`X11`、`X12`，`official-plus-practice`；`X11/X12` 只作反例。

### PRA010 用 Skill 承载可重复工作流

- **适用**：同一多步骤流程反复手工描述，或需要脚本、参考资料和模板。
- **做法**：一个 Skill 对应一个连贯用户意图；把详细资料放 references，把机械步骤放 scripts，入口只保留路由和关键流程。
- **放置**：跨项目放用户 Skill；项目专属放 `.agents/skills`。
- **验证**：换一个同类任务仍能复用，且不需要复制完整流程到 prompt。
- **依据**：`O07`、`O08`，`official-confirmed`。

### PRA011 优先修触发描述，再改实现

- **适用**：Skill 应该触发却没有触发、多个 Skill 同时抢同一任务、用户必须每次点名。
- **做法**：明确 `name`、`description`、适用场景、非适用场景和关键边界；用正例与负例做触发测试。触发不稳定时先改描述和示例。
- **放置**：Skill frontmatter 和触发测试。
- **验证**：一组应触发/不应触发任务得到稳定路由，且没有新增冲突。
- **依据**：`O07`、`O08`，`official-confirmed`。

### PRA012 Skill 脚本像小型 CLI

- **适用**：Skill 每次临时重写脚本、输出位置不固定、失败时仍继续生成结论。
- **做法**：脚本可从命令行运行，输入参数清楚，stdout 确定，错误显式，输出写入已知路径；配套输入、命令和期望输出示例。
- **放置**：Skill `scripts/` 和测试。
- **验证**：相同 fixture 得到稳定结果；失败返回非零状态并阻止不可靠结论。
- **依据**：`O08`，`official-confirmed`。

### PRA013 录制稳定的重复操作，再转成可编辑 Skill

- **适用**：重复的浏览器或桌面流程很难只用文字准确描述。
- **做法**：当前环境支持 Record & Replay 时，录制一次受控流程，将其转成可检查、可编辑 Skill；先审查生成内容再复用。
- **放置**：Skill；不要把录制内容当作不可见自动化黑盒。
- **验证**：在隔离数据上重放成功，关键动作和数据外发边界可检查。
- **依据**：`X01`，`experimental`。使用前重新确认当前 Codex 表面是否提供该能力。

### PRA014 自治程度必须由清晰度和可验证性决定

- **适用**：Codex 遇到阻塞后自行扩大范围、替用户做产品决定，或可逆小事频繁停下确认。
- **做法**：只有目标匹配、实现范围有界、验证路径可用、动作可逆时才提高自治；产品、安全、隐私、凭据和不可逆决策保留给用户。
- **放置**：任务 prompt、项目协作协议或专用编排 Skill。
- **验证**：自治任务能说明进入门槛、停止条件、证明和需要用户决定的例外。
- **依据**：`O01`、`O05`、`X06`，`official-plus-practice`。

### PRA015 要求最小实现并保护无关范围

- **适用**：出现多余抽象、代码膨胀、无关注释或文件被修改、反馈后全量重写。
- **做法**：明确非目标和禁止改动；要求每一处差异可追溯到请求；完成后检查孤儿代码和无关 diff。
- **放置**：当前 prompt；跨任务反复出现后才进入项目 `AGENTS.md`。
- **验证**：差异规模与目标匹配，无关文件不变，简化后仍通过测试。
- **依据**：`X03`，`practice-supported`。

### PRA016 优先使用可组合、可观察的工具

- **适用**：Agent 依赖只能人工操作的界面，工具输出无法被下一步解析，重复手工搬运数据。
- **做法**：优先选择稳定 CLI、结构化 API 或 MCP；让工具输出可解析、错误明确，并允许 Agent 在反馈中迭代。
- **放置**：项目工具链、Skill scripts 或 MCP 配置。
- **验证**：工具可在隔离环境重复调用，结果和错误能进入下一步判断。
- **依据**：`O08`、`X04`，`official-plus-practice`。

### PRA017 不把“多代理”默认等同于更快

- **适用**：同一问题被拆成过多代理、代理之间重复探索或同时修改同一区域。
- **做法**：只有工作独立且协调成本低时并行；读密集工作优先并行，写密集工作先隔离目录/worktree；主代理负责合并结论。
- **放置**：任务编排或编排 Skill。
- **验证**：每个代理有独立输入输出，冲突和重复工作低于节省的时间。
- **依据**：`O03`、`X05`，`official-plus-practice`。

### PRA018 网络、权限和外部内容按最小边界处理

- **适用**：Skill 同时拥有开放网络和宽文件权限，外部页面内容直接影响命令，敏感数据可能离开本机。
- **做法**：缩小权限和网络域，把外部内容视为不可信输入，明确哪些数据可以外发；破坏性或高影响动作保留确认。
- **放置**：`config.toml`、MCP/插件策略、Skill 安全边界。
- **验证**：在无额外权限时仍能完成只读部分；外发、写入和删除动作有明确审批点。
- **依据**：`O05`、`O08`，`official-confirmed`。

### PRA019 用项目愿景和验收门槛约束自动维护

- **适用**：自动分类 issue、长期维护或多项目编排缺少产品方向，导致“能修”被误当成“应该修”。
- **做法**：先读取可维护的项目愿景、当前目标和不做范围；只把方向匹配、实现有界、高置信且可实时验证的项目交给自治流程。
- **放置**：项目 `VISION.md`/产品文档、项目 `AGENTS.md` 和维护 Skill。
- **验证**：每个自治任务能追溯到项目目标，并有拒绝或升级给用户的条件。
- **依据**：`X06`、`X07`，`practice-supported`。不要复制其中的具体权限或持续自治设置。

### PRA020 监督概念错误，审查假设而不只看语法

- **适用**：代码能运行但设计假设错误、模型没有暴露困惑或权衡、用户过度相信“测试通过”。
- **做法**：审查模型代表用户做出的假设、边界条件、权衡和概念正确性；让 Codex 列出关键假设，并把输出当作贡献者 PR，而不是最终权威。
- **放置**：review prompt、项目审查清单和测试设计。
- **验证**：关键假设有证据或明确不确定性，测试覆盖真实风险而非只覆盖语法路径。
- **依据**：`O05`、`X03`，`official-plus-practice`。

## 反例与过度承诺

### 复制一份热门 AGENTS.md 就能普遍提效

X 上有高传播帖子建议把某个 65 行模板直接复制到全局指令，并使用“领先 90%”“提效 90%”等表达。它可以作为灵感来源，不能作为普遍规范：

- 官方指令链存在全局、项目和嵌套作用域，近端规则会覆盖上层规则。
- 他人的项目、工具、风险偏好和验证方式不等于用户的真实环境。
- 规则应来自重复发生的问题和可执行项目事实。

因此把 `X11`、`X12` 标记为 `R` 反例，不把传播量当作正确性证据。

### 更多代理一定更快

具名实践者同时展示了并行的价值和协调失败。官方也明确提醒并行写任务会增加冲突。默认建议是：先判断独立性，读任务优先并行，写任务先隔离，再衡量协调成本。

### 宽权限等于高效率

社区编排示例可能使用持续自治、绕过审批或高权限参数。这些是特定维护者的风险选择，不是可移植规范。知识网络只吸收“目标有界、证据门槛、停止条件和验证闭环”，不复制权限配置。

## 更新规则

- 官方产品行为、命令、路径和功能可用性超过 30 天后，在用于高置信建议前重新核验。
- `OX` 新能力在进入正式建议前查找对应官方文档；没有文档时保持 `experimental`。
- `P/C` 实践超过 90 天、链接失效或产品表面发生明显变化时重新检查。
- 传播量只用于发现候选，不进入权威等级和推荐强度。
- 新来源必须绑定现有节点或说明为什么需要新节点；不要把帖子摘要无限追加成资料堆。
